Code style requirements
General style
* keep code explicit and boring
* prefer small, obvious methods
* avoid clever abstractions
* preserve current project structure unless a change is necessary for this feature
* do not introduce new frameworks or architectural layers
Comment style
Use minimal inline comments.
Requirement
Only include inline comments when they are truly helpful for:
* non-obvious reasoning
* surprising behavior
* constraints that are easy to violate
Avoid comments for:
* restating what the code obviously does
* narrating simple assignments
* redundant section headers inside small methods
Preferred comment density
* sparse
* targeted
* explanation-oriented, not narration-oriented

Example comment style
# Store due_date as ISO text in SQLite so it round-trips cleanly through the API.
due_date_value = task.due_date.isoformat() if task.due_date else None
Bad style to avoid:
# Set the due date value
due_date_value = task.due_date.isoformat() if task.due_date else None

Function design
* prefer direct, typed functions
* keep method responsibilities narrow
* avoid mixing storage translation logic with unrelated business logic
Naming
Use clear names that reflect behavior.
Examples:
* add_task_to_queue
* expected_token_count
* dependency_str
Avoid vague names like:
* process_task
* meta
* extra_info

Test style
* one test should validate one clear behavior cluster
* keep payloads readable
* prefer explicit assertions over indirect inference
* do not rely on test execution order
